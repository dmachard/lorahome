# LoRa Protocol Specification (Binary)

This document describes the binary transmission format and data encapsulation used between **Sensor Nodes** (ESP32-C3) and the **Gateway** (ESP32-C6).

The protocol is optimized to minimize radio frame size (within the constraints of LoRa's limited bandwidth) while ensuring data authentication and encryption.

---

## 1. Over-The-Air Frame Structure

Each packet transmitted over LoRa has the following binary layout:

| Offset (bytes) | Field | Size (bytes) | Description | Visibility |
|---|---|---|---|---|
| `0` | Header | `9` | Node identifier, sequence number, random session ID. Used as Authenticated Associated Data (AAD). | Plaintext |
| `9` | Payload | `43` | Encrypted `SensorPayload` structure. | Encrypted |
| `52` | Auth Tag | `8` | Message Authentication Code (MAC) generated by AES-GCM. | Plaintext |

**Total standard frame size:** $9 + 43 + 8 = 60 \text{ bytes}$.

---

## 2. Header Structure

The 9-byte header identifies the sender, prevents replay attacks, and provides the material to construct the AES-GCM Initialization Vector (IV):

```c
struct Header {
  uint8_t  node_id;    // Unique node identifier (1–15)
  uint32_t seq;        // Sequence number (incremented on every transmission)
  uint32_t random_id;  // Random session ID generated at boot by the ESP32 TRNG
};
```

*   **`node_id` (1 byte):** Allows the gateway to instantly identify the sending node and load its AES key.
*   **`seq` (4 bytes, Big Endian):** Packet counter. Used to prevent replay attacks.
*   **`random_id` (4 bytes, Big Endian):** Hardware entropy from `esp_random()`. Ensures IV freshness (nonce uniqueness) even if the node reboots and resets its `seq` counter to zero.

---

## 3. Encrypted Payload (`SensorPayload`)

The `SensorPayload` structure is **43 bytes** and uses the `packed` attribute to prevent any compiler-inserted alignment padding:

```cpp
struct SensorPayload {
  uint8_t count;               // Number of readings present (max 6)
  SensorReading readings[6];   // Static array of readings (6 × 5B = 30B)
  uint8_t reset_reason;        // Last ESP32 reset reason
  uint8_t error_code;          // System error code (0 = OK)
  uint16_t tx_interval;        // Transmission interval (seconds)
  char name[8];                // Node name (8 chars, null-terminated)
} __attribute__((packed));
```

### `SensorReading` detail (5 bytes each):

```cpp
struct SensorReading {
  uint8_t type;   // Sensor/reading type identifier (1 byte)
  int32_t value;  // Signed 32-bit integer raw value (4 bytes)
} __attribute__((packed));
```

---

## 4. Sensor Type Identifiers (`ReadingType`)

Reading types are standardized via the following enumeration:

| Value | Enum Name | Sensor | Physical Quantity | Gateway Scaling Rule |
|---|---|---|---|---|
| `1` | `TYPE_DHT22_TEMP` | DHT22 | Temperature | raw / 100.0f (°C) |
| `2` | `TYPE_DHT22_HUM` | DHT22 | Humidity | raw / 100.0f (%) |
| `3` | `TYPE_AHT20_TEMP` | AHT20 | Temperature | raw / 100.0f (°C) |
| `4` | `TYPE_AHT20_HUM` | AHT20 | Humidity | raw / 100.0f (%) |
| `5` | `TYPE_BMP280_TEMP` | BMP280 | Temperature | raw / 100.0f (°C) |
| `6` | `TYPE_BMP280_PRES` | BMP280 | Atmospheric Pressure | raw / 10.0f (hPa) |
| `7` | `TYPE_BH1750_LUX` | BH1750 | Light Intensity | raw (Lux) |
| `8` | `TYPE_BATTERY` | Internal | Battery Voltage | raw (mV) |

---

## 5. Value Scaling Convention

To avoid transmitting floating-point numbers (which are bandwidth-expensive and architecturally inconsistent), physical values are multiplied by a fixed factor before sending and divided back by the gateway:

*   **Temperatures:** Sent as `float × 100` (e.g., $23.45 \text{ °C} \rightarrow 2345$).
*   **Humidity:** Sent as `float × 100` (e.g., $45.67 \text{ \%} \rightarrow 4567$).
*   **Pressure:** Sent as `float × 10` (e.g., $1013.25 \text{ hPa} \rightarrow 10132$, i.e., Pa / 10).
*   **Lux & Battery Voltage:** Sent as raw integers with no scaling (e.g., $4200 \text{ mV} \rightarrow 4200$).
